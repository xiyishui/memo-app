# 项目总结报告

## Memo App - 个人备忘录应用

---

## 一、项目概述

本次实训项目采用 Next.js 16 + Supabase 技术栈，开发了一个功能完整的个人备忘录与待办事项管理应用。项目历时3天，涵盖用户认证、便签管理、待办事项、标签分类、收藏置顶、回收站、最近查看、头像设置等完整功能链，并成功部署到 Vercel 线上平台。

---

## 二、遇到的最大困难及解决过程

### 2.1 数据库迁移：从本地文件到云端

项目初期采用本地 JSON 文件存储数据，这种方式在开发阶段简单直接，但存在明显的局限性：数据无法持久化、多设备无法同步、无法部署到线上。Day 3 决定迁移到 Supabase（PostgreSQL）云端数据库时，遇到了整个项目中最棘手的挑战。

**第一个问题是路由迁移遗漏。** 项目中有8个 API 路由文件，大部分已改为 Supabase 查询，但 memos/[id]/route.js（便签详情/更新/删除）和 	rash/route.js（回收站列表）仍然保留着旧的本地文件读写逻辑。这导致便签删除后不会真正进入回收站，回收站页面也一直为空。排查过程花费了不少时间，最终通过逐一比对每个 API 路由的代码才定位到问题。

**第二个问题是 PostgREST Schema 缓存。** 在 Supabase 中创建表后，PostgREST 的 schema 缓存不会自动刷新，导致查询时报 "Could not find the 'createdat' column of 'memos' in the schema cache" 错误。起初以为是建表语句有问题，反复检查了多次 SQL，甚至重新建表。后来查阅 Supabase 文档才发现需要执行 NOTIFY pgrst, 'reload schema' 来刷新缓存。

**第三个问题是中文乱码。** 项目运行后发现便签列表、登录页面、待办事项等几乎所有页面的中文都显示为乱码（如"璇峰厛鐧诲綍"这样的奇怪字符）。排查发现这是因为在 PowerShell 环境下使用 Set-Content 写入文件时，默认编码与 UTF-8 不一致，导致整个项目的 JS 文件中文字节被破坏。最终逐文件重写，用 .NET 的 System.IO.File.WriteAllText 指定 UTF-8 编码才彻底解决。

### 2.2 部署的网络限制

项目开发环境无法直接连接 GitHub 和 Vercel，所有推送都需要手动操作。但通过分步策略——在本地完成开发后，将 commit 打包，在自己电脑上推送到 GitHub，再由 Vercel 自动同步部署——最终成功上线。

## 三、对AI辅助编程的心得体会

### 3.1 效率提升是实实在在的

在传统开发模式下，3天时间从零搭建一个包含前后端完整功能的应用几乎不可能。而借助 AI 编码助手，大幅减少了以下环节的时间消耗：

- **样板代码生成**：API 路由的 try-catch 结构、前端组件的 useState/useEffect 模式、CSS 样式等重复性工作，由 AI 高效完成
- **问题排查**：遇到 Schema cache 这类不熟悉的 Supabase 特定问题时，AI 能快速定位原因并提供解决方案
- **代码重构**：从本地文件迁移到 Supabase 涉及 8 个路由文件的重写，AI 能理解整体架构并保持一致风格

### 3.2 AI 不是替代，而是协作

这次实践让我深刻体会到，AI 编程助手不是一个"自动完成"工具，而是一个"协作伙伴"。AI 擅长处理已知模式、快速生成代码、进行模式化重构，但在以下方面仍然需要人的判断：

- **架构决策**：选择什么技术栈、数据如何组织、什么时候引入依赖——这些需要开发者的经验判断
- **问题定位**：当出现多个错误叠加时（比如乱码 + 数据库问题同时出现），AI 倾向于逐个排查，而人可以通过直觉先缩小范围
- **代码质量把控**：AI 生成的代码不一定是最优解，需要进行 Code Review 和优化

### 3.3 提示工程的价值

我发现 Prompt 的质量直接影响 AI 的输出质量。模糊的指令（如"帮我完善"）得到的结果往往需要多次迭代修正，而具体明确的指令（如"在注册页面添加确认密码输入框，两次密码一致才能注册"）能让 AI 一次给出接近最终结果的代码。这也反向锻炼了我自己的需求拆解能力——要想让 AI 理解我想要的，首先我得自己想清楚。

## 四、个人在项目中的具体贡献

### 4.1 需求分析与架构设计

- 根据课程考核要求，将项目拆分为3天的开发计划
- 决定采用 Next.js 16 的 App Router 架构，利用其 API Routes 实现前后端一体化
- 选择 Supabase 作为数据库后端，利用其免费额度实现线上部署
- 设计了 5 张数据库表（users、memos、todos、trash、recent）的表结构和关联关系

### 4.2 核心功能开发

- 实现了用户注册/登录系统，包括密码加密存储、Token 验证、数据隔离
- 开发了便签的增删改查功能，包括标签分类、收藏置顶、搜索筛选
- 实现了待办事项的完整 CRUD，支持编辑模式和完成状态切换
- 设计了回收站机制（软删除 + 还原）
- 实现了最近查看记录功能
- 开发了个人中心页面，包含头像设置、数据统计

### 4.3 问题排查与修复

- 排查并修复了 memos/[id]/route.js 和 trash/route.js 未迁移到 Supabase 的问题
- 解决了 Supabase Schema cache 导致的数据插入失败问题
- 系统性地修复了 13 个文件的中文乱码问题
- 修复了 memoId 与 memoid 参数名大小写不一致的 bug
- 统一了 API 的错误响应格式

### 4.4 文档与代码审查

- 编写了完整的 README.md（项目介绍、技术栈、安装指南）
- 编写了 API 文档（docs/API.md），包含所有接口的请求/响应格式
- 进行了代码审查（docs/CODE_REVIEW.md），识别并记录了 12 个改进点
- 编写了 Prompt 日志（docs/prompt_log.md），记录开发过程中的关键对话

### 4.5 部署与上线

- 配置了 Supabase 环境变量
- 完成了 Git 版本管理和 GitHub 推送
- 实现了 Vercel 自动化部署

## 五、总结与展望

这次项目实践让我完整经历了一个全栈应用从设计、开发、调试到部署的全流程。最大的收获不是学会了某个具体的技术，而是体验了"人机协作"这种新的开发模式——AI 处理细节和重复劳动，人把握方向和做出判断。

在未来的项目中，我会更多关注代码质量和安全性（如密码加盐、XSS 防护），并尝试引入 TypeScript 来提升代码的健壮性。AI 辅助编程不是终点，而是起点——它让我能把更多精力放在"做什么"和"为什么做"上，而不是"怎么做"上。
